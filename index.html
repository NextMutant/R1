Daily Routine (45 min)

1 problem cold (no hints) → review approach → implement → test edge cases → read 1 top discussion. Never skip the discussion — it's where you learn patterns.

Weekly Mock (90 min)

Every Sunday: timed 3-problem LeetCode contest simulation. Review where you got stuck. Pattern-gap analysis: which patterns are still slow?

FAANG Interview Standard

Talk through your approach before coding. Ask clarifying questions. Write brute force first, then optimize. State time + space complexity. Test with examples and edge cases aloud.

Speed Benchmarks

Easy: <8 min. Medium: <20 min. Hard: understand approach within 15 min, implement in 30. By end of W6, you should solve 3 mediums in 45 mins total.

System Design Practice Plan
System design is the highest-signal interview category at senior levels. Learn the framework first — then apply it to 6 classic systems. Each design session: 45 minutes, whiteboard format.
THE DESIGN FRAMEWORK (use every time)
1. Requirements
  • Functional requirements
  • Non-functional (scale, latency)
  • Back-of-envelope estimates
2. API Design
  • REST endpoints
  • Request/response shapes
  • Rate limiting strategy
3. Data Model
  • Entities and relationships
  • SQL vs NoSQL choice
  • Schema design
4. High-Level Design
  • Client → LB → App → DB
  • Caching layer placement
  • CDN usage
5. Deep Dive
  • Bottleneck identification
  • Scaling the hot path
  • Failure modes + recovery
6. Trade-offs
  • What you'd change with 10x scale
  • Consistency vs availability
  • Cost vs complexity

  • Presence system (Redis pub/sub)
  • Message storage + pagination
4. Ride-Sharing (Uber)
  • Geospatial indexing
  • Driver matching algorithm
  • Real-time location updates
  • Surge pricing system
5. Payment System
  • Idempotency keys
  • Two-phase commit

FINE-TUNING DECISION FRAMEWORK
USE PROMPT ENGINEERING WHEN

Task is generic, base model handles it well, you need fast iteration, cost matters, problem is solvable with better instructions.

USE RAG WHEN

You need up-to-date knowledge, domain-specific documents, verifiable citations, dynamic knowledge base, or long-context retrieval.

FINE-TUNE WHEN

Specific format/style needed at scale, sensitive data can't go to external API, task requires deep domain syntax, RAG is too slow.
